# Backpressure saves your LLM service when a downstream API throttles you

A queue in front of a rate-limited model API lets your frontend stay responsive while you absorb bursts, instead of fanning out unlimited concurrent calls and getting 429s.

When a single request fans out to several tool calls, each hitting the same provider, you can exhaust the rate limit in milliseconds. Wrap the calls in a bounded concurrency limiter so in-flight work is capped and excess waits instead of failing.

```ts
// p-limit style bounded concurrency around a rate-limited LLM call
class Semaphore {
  private permits: number;
  private queue: (() => void)[] = [];
  constructor(n: number) { this.permits = n; }
  async acquire(): Promise<() => void> {
    if (this.permits > 0) { this.permits--; return () => this.release(); }
    return new Promise(res => this.queue.push(res));
  }
  private release() {
    const next = this.queue.shift();
    if (next) next(); else this.permits++;
  }
}

const limiter = new Semaphore(5); // max 5 concurrent model calls

export async function safeModelCall(prompt: string) {
  const done = await limiter.acquire();
  try {
    return await callProvider(prompt); // may still throw 429, retry here
  } finally {
    done();
  }
}
```

- Cap concurrency to a value below your provider's published RPM/TPM, not the theoretical max.
- Pair the limiter with a short retry and exponential backoff so a transient 429 becomes a delay, not an error.
- Measure queue depth in your metrics. A growing backlog means the limit is too low for your traffic, so raise it or shed load.
