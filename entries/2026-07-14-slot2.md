# Bounded concurrency beats Promise.all for LLM batches

Fanning out hundreds of LLM calls with a bare `Promise.all` will trip provider rate limits and spike your memory. A small concurrency cap with exponential-backoff retries keeps throughput high without 429 storms or a failed batch.

```ts
// Bounded-concurrency LLM batch runner: token-paced, per-item retry.
async function callLLM(prompt: string): Promise<string> {
  const r = await fetch(MODEL_URL, {
    method: "POST",
    body: JSON.stringify({ prompt }),
  });
  if (r.status === 429) throw new Error("rate_limited");
  return r.text();
}

async function withRetry<T>(fn: () => Promise<T>, tries = 3): Promise<T> {
  for (let i = 0; i < tries; i++) {
    try {
      return await fn();
    } catch (e) {
      if (i === tries - 1) throw e;
      await new Promise((r) => setTimeout(r, Math.min(1000 * 2 ** i, 8000)));
    }
  }
  throw new Error("unreachable");
}

import PQueue from "p-queue";
const q = new PQueue({ concurrency: 8 }); // cap in-flight calls

export function runBatch(prompts: string[]): Promise<string[]> {
  return Promise.all(prompts.map((p) => q.add(() => withRetry(() => callLLM(p)))));
}
```

- Concurrency 8 with exponential backoff caps memory and avoids 429 storms, raise it only after checking your provider's RPM limit.
- A bare `Promise.all` over the array retries nothing and fails the whole batch on one bad item.
- Per-item retry recovers from transient 429s without aborting the run, so partial failures stay isolated.
