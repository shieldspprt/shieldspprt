# Cache LLM tool-call schemas at the gateway, not per request

If every inbound request rebuilds the same tool definitions before calling the model, you pay a serialization and allocation tax on the hot path, and you risk schema drift between requests. Build the schema once, cache it, and reuse.

```ts
// tools.ts: constructed once at boot, not per request
import { zodToJsonSchema } from "zod-to-json-schema";
import { z } from "zod";

const searchSchema = z.object({
  query: z.string(),
  topK: z.number().int().min(1).max(50),
});

// Build the model-facing tool list a single time.
export const TOOL_DEFS = [
  {
    type: "function",
    function: {
      name: "search_docs",
      description: "Semantic search over the docs index.",
      parameters: zodToJsonSchema(searchSchema),
    },
  },
];

// chat.ts: request handler reuses the cached defs
export async function handleChat(req: ChatRequest) {
  const res = await model.chat({
    messages: req.messages,
    tools: TOOL_DEFS, // no per-request rebuild
  });
  return res;
}
```

Takeaways:
- Construct tool schemas at startup, not inside the request handler, so CPU and GC pressure stay off the latency-critical path.
- Pin a single schema source so every call sends identical JSON, which keeps model behavior stable and makes eval diffs meaningful.
- Only rebuild when a tool definition actually changes, and then invalidate the cached object rather than mutating it in place.
