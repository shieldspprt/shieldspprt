# Cache LLM tool-call schemas at the gateway, not per request

Validating and dispatch of agent tool calls gets expensive when every request re-parses large JSON schemas. Build the schema registry once at process start, then serve a frozen lookup from memory.

A common mistake is parsing the tool manifest inside the request handler. Under bursty agent traffic that repeats the same work thousands of times per minute and adds tail latency that looks like a model slowdown.

```ts
// tools.ts, loaded once at boot
import { z } from "zod";

const ToolSchemas = {
  search: z.object({
    query: z.string().min(1).max(500),
    top_k: z.number().int().min(1).max(50).default(5),
  }),
  // ...other tools
} as const;

export type ToolName = keyof typeof ToolSchemas;

// Validate a raw model payload against the cached schema
export function parseToolCall(name: string, raw: unknown) {
  const schema = ToolSchemas[name as ToolName];
  if (!schema) throw new Error(`unknown tool: ${name}`);
  return schema.parse(raw); // throws ZodError with field-level detail
}
```

In the request path you only call `parseToolCall`, which is O(1) schema lookup plus a parse. No JSON.parse of the manifest, no filesystem read, no rebuild of the validator.

- Keep the registry immutable and shared across workers so each process loads it exactly once.
- Return the ZodError message straight to the model, it is already field-specific and teaches the next retry.
- If tools change at runtime, version the registry key and hot-swap the object, do not mutate the live one.
