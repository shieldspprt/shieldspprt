# Backend idempotency keys stop duplicate writes under retry storms

A client that retries on timeout will re-send the same request. Without an idempotency key, a retried POST can create a second record, a second charge, or a second email. Store the key once and short-circuit repeats.

```ts
// Hono API route with idempotency
const seen = new Map<string, Response>(); // swap for Redis in prod

export default async (c: Context) => {
  const key = c.req.header("Idempotency-Key");
  if (!key) return c.json({ error: "Idempotency-Key required" }, 400);

  const cached = await kv.get(`idem:${key}`);
  if (cached) return new Response(cached, { headers: { "X-Replayed": "1" } });

  const result = await createOrder(await c.req.json()); // side effect
  const body = JSON.stringify(result);
  await kv.set(`idem:${key}`, body, { ex: 86400 }); // 24h TTL
  return c.body(body);
};
```

- The key is generated by the caller and sent on every retry, so the server detects duplicates even across crashed instances.
- Persist the key with a TTL so storage stays bounded, and return the original response on replay rather than re-running the side effect.
- For multi-step flows, combine the key with a status column (pending, done) so a half-finished write can be resumed or skipped atomically.
