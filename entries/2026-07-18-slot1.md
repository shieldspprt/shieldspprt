# Cache your LLM responses at the edge to cut cost and latency

When the same prompt is asked repeatedly, serve the model output from a cheap key-value store keyed on a hash of the request, instead of paying for a new inference every time. This works especially well for stable content like summaries, classifications, and embeddings.

```ts
// Cloudflare Worker: dedupe identical LLM calls
const MODEL = "gpt-4o-mini";

async function chat(messages: {role: string; content: string}[]) {
  const res = await fetch("https://api.openai.com/v1/chat/completions", {
    method: "POST",
    headers: {
      "content-type": "application/json",
      authorization: `Bearer ${API_KEY}`,
    },
    body: JSON.stringify({ model: MODEL, messages }),
  });
  return (await res.json()).choices[0].message.content;
}

export default {
  async fetch(req: Request, env: { KV: any; API_KEY: string }) {
    const body = await req.json();
    const key = await crypto.subtle.digest(
      "SHA-256",
      new TextEncoder().encode(JSON.stringify(body.messages)),
    ).then((b) => [...new Uint8Array(b)].map(x => x.toString(16)).join(""));

    const cached = await env.KV.get(key);
    if (cached) return new Response(cached, { headers: { "x-cache": "HIT" } });

    const out = await chat(body.messages);
    await env.KV.put(key, out, { expirationTtl: 86400 });
    return new Response(out, { headers: { "x-cache": "MISS" } });
  },
};
```

- Key the cache on a normalized hash of the inputs, not the raw JSON, so whitespace or key-order differences do not create duplicate entries.
- Set a TTL so facts can refresh. Use a short TTL for volatile data, a long one for stable reference content.
- This pattern moves the cost from per-request model inference to per-hit KV reads, often a 100x reduction at the edge.
