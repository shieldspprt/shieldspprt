# Design idempotent webhook handlers with a dedupe key

A webhook can arrive more than once. Payment providers, queue redeliveries, and load balancer retries all replay events, so your handler must produce the same effect on the second delivery as the first.

The simplest robust pattern is a dedupe table keyed by the event id, checked inside a transaction before you do any work:

```ts
// Bun + SQLite, inside a single transaction
async function handleWebhook(eventId: string, payload: any) {
  const tx = db.transaction((id: string, body: any) => {
    const seen = db
      .query("SELECT 1 FROM webhook_events WHERE event_id = ?")
      .get(id);
    if (seen) return { status: "duplicate" };

    db.run(
      "INSERT INTO webhook_events (event_id, created_at) VALUES (?, ?)",
      [id, Date.now()]
    );

    // Only now mutate real state, e.g. mark order paid
    db.run("UPDATE orders SET status = 'paid' WHERE id = ?", [
      body.orderId,
    ]);
    return { status: "processed" };
  });

  return tx(eventId, payload);
}

// HTTP layer: return 200 even for duplicates so the
// provider stops retrying, but do not re-run side effects.
const result = await handleWebhook(c.req.header("x-event-id")!, await c.req.json());
return c.json(result, 200);
```

- Key dedupe on the provider's event id, not on your internal row id, because the duplicate arrives with the same external id and a fresh internal one.
- Wrap the check and the state change in one transaction so a crash between the two steps cannot create a duplicate, and so a concurrent redelivery is serialized by the DB lock.
- Always return 2xx for duplicates, since a 4xx or 5xx tells the sender to keep retrying and you get more replays, not fewer.
