# Retry LLM calls with jittered exponential backoff, not fixed sleeps

Fixed `sleep(1000)` retries pile up on shared rate limits and turn a brief 429 into a thundering herd. Cap the attempts, double the delay each try, and add jitter so many clients do not fire at once.

```ts
async function callWithRetry<T>(
  fn: () => Promise<T>,
  maxAttempts = 5,
): Promise<T> {
  let attempt = 0;
  while (true) {
    try {
      return await fn();
    } catch (err) {
      const isRateLimit =
        err?.status === 429 || err?.code === "rate_limit_exceeded";
      if (!isRateLimit || attempt >= maxAttempts - 1) throw err;

      // base: 250ms, 500ms, 1s, 2s ... then jitter up to +50%
      const base = 250 * 2 ** attempt;
      const jitter = Math.random() * base * 0.5;
      const wait = Math.min(base + jitter, 8000);

      // honor a server-supplied backoff hint when present
      const retryAfter = Number(err?.headers?.["retry-after"]);
      await sleep(Number.isFinite(retryAfter) ? retryAfter * 1000 : wait);
      attempt++;
    }
  }
}

const sleep = (ms: number) => new Promise((r) => setTimeout(r, ms));
```

- Read `Retry-After` and respect it, otherwise you waste an attempt against a window that has not opened yet.
- Cap total wait (here 8s) so a bad upstream does not hang the request path for minutes.
- Jitter prevents synchronized retries across many workers, which is what actually causes a 429 to spread.
