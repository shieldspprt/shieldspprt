# Colocate compute with data to avoid shipping rows you never render

A backend that filters and paginates inside the database returns a few rows over the wire, while a backend that loads a table into the app and slices it in memory pays for bandwidth, memory, and latency on every request.

The rule of thumb: let the database do what it is good at. Filtering, aggregation, and windowed pagination belong in SQL, not in a fetch-and-loop in your service. This keeps the response small and the query planner free to use indexes.

```ts
// Before: pull everything, then paginate in the app
const rows = await db.query("SELECT id, title, created_at FROM posts");
const page = rows.slice(offset, offset + limit); // loads all rows every call

// After: push pagination into the query
const page = await db.query(
  `SELECT id, title, created_at
   FROM posts
   WHERE created_at < $1          -- keyset cursor, index-friendly
   ORDER BY created_at DESC, id DESC
   LIMIT $2`,
  [cursor ?? new Date().toISOString(), limit]
);
```

This uses a keyset (seek) cursor instead of `OFFSET`, so deep pages stay fast because the planner starts the scan at the cursor, not at row zero.

- Filter and paginate in SQL so the network and your app only ever touch the rows you return.
- Prefer keyset pagination (`WHERE ... < cursor`) over `OFFSET` for stable, index-backed scrolling on large tables.
- Keep the selected columns narrow. Selecting `*` can pull blobs and JSON you never render, bloating both the query and the payload.
