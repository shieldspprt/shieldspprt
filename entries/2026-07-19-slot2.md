# Ship a typed tool layer so agents never guess a schema

A thin typed wrapper around your agent tools makes the model's inputs verifiable at the boundary, and it turns silent contract drift into a thrown error instead of a bad tool call.

```ts
// Tool defines its own input schema and a typed run function.
interface Tool<I, O> {
  name: string;
  schema: ZodSchema<I>;          // validates model-supplied args
  run: (input: I) => Promise<O>;
}

async function callTool<I, O>(tool: Tool<I, O>, raw: unknown): Promise<O> {
  const parsed = tool.schema.safeParse(raw);
  if (!parsed.success) {
    throw new Error(`tool ${tool.name}: ${parsed.error.message}`);
  }
  return tool.run(parsed.data);
}

// Example: a typed search tool the model can invoke.
const searchDocs: Tool<{ query: string; topK: number }, string[]> = {
  name: "searchDocs",
  schema: z.object({ query: z.string(), topK: z.number().int().min(1).max(20) }),
  run: async ({ query, topK }) => retrieve(query, topK),
};

// If the model sends topK: "5" or omits query, callTool throws before run().
```

Takeaways:
- Validate the model's tool arguments with a schema before executing. A bad arg becomes a clear error, not a confusing runtime failure deep in your service.
- Keep the schema co-located with the tool implementation so the contract drifts in one place, not across a prompt and a handler.
- Typed tools double as documentation. The schema is the single source of truth for what the agent is allowed to call.
