# Backend idempotency keys stop duplicate writes from retries

A single network retry can hit your API twice. Put the idempotency key in the database, not just in memory, so a restarted or second instance never double-writes the same request.

```ts
// Bun + Postgres (via postgres.js)
import postgres from "postgres";
const sql = postgres(process.env.DATABASE_URL!);

// Returns the existing row if the key was already processed.
async function handleOrder(key: string, payload: Order) {
  // Unique constraint on idempotency_key makes the second insert fail.
  const existing = await sql`
    INSERT INTO processed_orders (idempotency_key, payload, status)
    VALUES (${key}, ${JSON.stringify(payload)}, 'done')
    ON CONFLICT (idempotency_key) DO NOTHING
    RETURNING *;
  `;
  if (existing.length > 0) {
    return { ok: true, id: existing[0].id, replay: false };
  }
  const prior = await sql`
    SELECT * FROM processed_orders WHERE idempotency_key = ${key};
  `;
  return { ok: true, id: prior[0].id, replay: true };
}

// Client sends the same key on every retry:
// POST /orders  { "Idempotency-Key": "f47ac10b-...", "item": "x" }
```

- The `ON CONFLICT DO NOTHING` pattern uses a unique index, so correctness survives process crashes and horizontal scaling, not just a single in-memory map.
- Generate the key on the client per logical action, and have it travel through every retry so the fifth attempt is still a replay, not a new order.
- Pair this with the request-level idempotency key check at the gateway so you reject replays fast, before doing any work.
