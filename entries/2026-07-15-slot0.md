# Order a composite Postgres index to match your query shape

A multi-tenant query that filters on `tenant_id` and `status` then sorts by `created_at` will scan and re-sort unless the index column order mirrors that exact access path. Lead with the equality columns, put the range or sort column last.

```sql
-- Bad: two separate single-column indexes. The planner still sorts,
-- and only one can drive the seek efficiently.
CREATE INDEX orders_tenant ON orders (tenant_id);
CREATE INDEX orders_created ON orders (created_at);

-- Good: equality columns first, sort column last. One index serves
-- both the WHERE and the ORDER BY, so no sort step is needed.
CREATE INDEX orders_tenant_status_created
  ON orders (tenant_id, status, created_at DESC);

-- The hot query: newest OPEN orders for one tenant.
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, amount, created_at
FROM orders
WHERE tenant_id = 't_42' AND status = 'OPEN'
ORDER BY created_at DESC
LIMIT 25;
-- Index Scan using orders_tenant_status_created,
-- no Sort node, rows returned in index order.
```

- Equality filters (`tenant_id`, `status`) must come before the sort/range column (`created_at`) or the index cannot satisfy the `ORDER BY`.
- Add `DESC` to match your sort direction. A mismatched direction forces a sort even when the columns are correct.
- `EXPLAIN (ANALYZE, BUFFERS)` is the only way to confirm a Sort node disappeared. A missing sort in the plan is your proof the index is doing its job.
