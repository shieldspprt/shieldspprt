# Idempotent API endpoints with an idempotency key

Retries are normal in distributed systems: a flaky network or a double-click can fire the same POST twice. Without idempotency, that means two charges, two emails, or two rows. An idempotency key lets the server return the first result instead of repeating the work.

## Backend (Bun / Hono)

```ts
import { Hono } from "hono";

const app = new Hono();
const seen = new Map<string, Response>();

app.post("/charge", async (c) => {
  const key = c.req.header("Idempotency-Key");
  if (!key) return c.json({ error: "Idempotency-Key required" }, 400);

  if (seen.has(key)) return seen.get(key)!;

  const result = await chargeCard(await c.req.json());
  const res = c.json(result, 201);
  seen.set(key, res);
  return res;
});
```

In production, store the key and the response in a database (Postgres or Redis) with a TTL so the guarantee survives restarts and holds across instances. The client just sends the same UUID on retry.

## Why it matters

Payments, signups, and orders are the flows recruiters care about most. Designing for safe retries signals you have shipped real systems, not toy demos.
