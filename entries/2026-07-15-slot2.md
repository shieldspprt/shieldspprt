# A prompt cache turns repeat questions into instant, free hits

Every identical prompt that reaches your LLM is a paid, slow round trip you have already paid for. Key the request on a normalized hash and serve the stored answer on a second hit. This is app-side caching, separate from the provider's own prompt cache, and it also shields you from runaway retry loops.

```ts
// A prompt cache keyed on a normalized hash turns repeat questions into
// instant hits. Cheap to add, and it protects you when a retry loop fires
// or a user re-asks the same thing.
import { createHash } from "node:crypto";

function cacheKey(model: string, system: string, user: string): string {
  const norm = (s: string) => s.trim().replace(/\s+/g, " ").toLowerCase();
  return createHash("sha256")
    .update(`${model}|${norm(system)}|${norm(user)}`)
    .digest("hex");
}

const store = new Map<string, { answer: string; ts: number }>();
const TTL_MS = 1000 * 60 * 60; // one hour

export async function askWithCache(model: string, system: string, user: string) {
  const key = cacheKey(model, system, user);
  const hit = store.get(key);
  if (hit && Date.now() - hit.ts < TTL_MS) return hit.answer; // cache hit

  const answer = await llm.complete({ model, system, user });
  store.set(key, { answer, ts: Date.now() });
  return answer;
}
```

- Normalize whitespace and case before hashing. Otherwise "What is 2+2?" and "what is 2 + 2 ?" miss each other and you pay twice.
- Set a TTL. Models and prompts change, so a stale answer is worse than a fresh, paid call. Size the window to how fast your answers go stale.
- Exact-hash caching is the 80 percent solution. Add an embedding-similarity lookup when near-duplicate prompts are common, but start with the hash.
