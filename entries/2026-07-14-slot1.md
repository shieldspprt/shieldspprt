# Idempotency keys keep LLM retries from double-firing

A network timeout on an AI provider call is ambiguous: you do not know if the model already generated. Wrap outbound calls with a short-lived idempotency key so a retried request either returns the cached response or is skipped, instead of producing a duplicate completion, duplicate charge, or duplicate side effect.

```ts
// Hono API route, backed by Redis. Same key within the TTL returns the first result.
import { Redis } from "ioredis";
const redis = new Redis(process.env.REDIS_URL!);

export async function callLLM(c: Context) {
  // Stable key from the request body, not a random uuid, so retries match.
  const body = await c.req.json();
  const key = `llm:${crypto.createHash("sha256").update(JSON.stringify(body)).digest("hex")}`;

  const cached = await redis.get(key);
  if (cached) return c.json(JSON.parse(cached), 200, { "X-Cache": "HIT" });

  // Claim the key with a short TTL before the slow call to block concurrent dupes.
  const claimed = await redis.set(key, "PENDING", "NX", "EX", 120);
  if (!claimed) return c.json({ error: "duplicate in flight" }, 409);

  try {
    const result = await provider.generate(body); // may time out
    await redis.set(key, JSON.stringify(result), "EX", 3600);
    return c.json(result);
  } catch (err) {
    await redis.del(key); // free the key so a true retry can proceed
    throw err;
  }
}
```

- Hash the request payload into the key so identical retries collapse, but different prompts get distinct keys.
- Use `SET key val NX EX` to claim the slot before the call. The failing path deletes the key so only genuine retries pay the cost, not every future identical prompt.
- Pair this with an exponential backoff plus jitter on the client side so you do not hammer a 429 from the provider.

This pattern is what keeps a flaky provider from billing you twice or sending a user two chatbot answers for one click.
